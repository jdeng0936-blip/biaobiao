## 一、图片库

### 📍 当前状态：❌ 未实现

当前系统只处理文本（PDF/Word/TXT 中的文字内容），**没有图片管理模块**。

### 🔧 需要建设什么？

技术标书中的图片需求非常明确——可以预分类：

| 图片分类 | 举例 | 管理方式 |
|---|---|---|
| **施工工艺图** | 钢筋绑扎节点图、模板支撑示意图 | 按工艺节点分类标签 |
| **施工流程图** | 施工工序流程、质量控制流程 | 按章节类型关联 |
| **平面布置图** | 施工现场临设布置 | 按项目类型模板化 |
| **进度计划图** | 甘特图、横道图 | 按工期动态生成 |
| **机械设备图** | 、塔吊、泵车照片 | 按设备型号索引 |
| **安全文明图** | 安全标识、围挡效果图 | 通用素材库 |
| **组织机构图** | 项目部组织架构图 | 按人员配置动态渲染 |

### 🏗 建设方案（建议 v0.4 迭代）

```
1. 新建 image_library 模块
   ├── 上传接口：支持 PNG/JPG/CAD 导出图
   ├── 标签系统：工艺类型 + 工程类型 + 章节关联
   ├── 搜索：按标签/关键词检索
   └── AI 自动插图：生成章节时自动匹配推荐图片

2. 数据模型
   ├── image_id (UUID)
   ├── filename / url
   ├── category (施工工艺/流程/布置/安全)
   ├── tags ["钢筋", "绑扎", "节点"]
   ├── applicable_sections ["施工方案", "质量保证"]
   └── tenant_id (租户隔离)

3. AI 集成
   ├── 章节生成时 → 检索匹配图片 → 插入占位符
   └── Word 导出时 → 占位符替换为真实图片
```

### 📊 初期需要多少图片？

| 工程类型 | 建议图片量 | 覆盖范围 |
|---|---|---|
| 市政道路 | **80~120 张** | 路基、路面、排水、交安、绿化 |
| 房建工程 | **100~150 张** | 基坑、结构、装饰、机电、消防 |
| 水利工程 | **60~80 张** | 围堰、浇筑、防渗、闸门 |
| 通用素材 | **50~80 张** | 安全文明、环保、组织架构模板 |
| **合计初期** | **~300 张** | 覆盖核心工艺节点 |

## 二、初期测试需要的数据体量

### 📍 最小可用数据集（MVP Test）

| 类别 | 最低量 | 推荐量 | 说明 |
|---|---|---|---|
| **中标范文** | 3 份 | 10 份 | 同类型项目的完整技术标 |
| **招标文件** | 1 份 | 3 份 | 含评分标准的完整招标文件 |
| **行业规范** | 0 份 | 5 份 | GB/T、JTG 等核心规范 |
| **企业资料** | 1 份 | 3 份 | 公司简介、业绩、资质 |
| **施工工艺** | 系统内置 248 个 | — | 工艺图谱已包含 |
| **知识切片总量** | ~500 条 | ~3000 条 | 系统自动切片 |

### 💡 测试阶段建议

```
第一周：投喂 3 份中标范文 + 1 份招标文件
       → 测试基本生成流程是否通畅

第二周：增加到 10 份范文 + 3 份规范
       → 对比生成质量提升幅度

第三周：加入企业资料 + 模拟多用户
       → 测试多租户隔离 + 个性化生成
```

## 三、系统自我迭代能力

### 📍 当前已实现：✅ 数据飞轮（被动进化）

```
用户反馈 → 飞轮机制 → 知识回灌
   ↓
✅ 采纳 → 原文标记为"优质语料"→ 提升类似内容生成权重
✏️ 修改 → 记录原文+修改版 → AI 学习修改偏好（diff_ratio）
❌ 拒绝 → 标记为"低质量"→ 降低类似输出概率
   ↓
flywheel_sunk 指标 → 已回灌到知识库的高质量语料数量
```

**这是被动进化**——依赖用户操作触发。

### 🔮 未来可建设：主动进化

| 能力 | 当前 | 目标 |
|---|---|---|
| 用户反馈学习 | ✅ 已有 | 持续优化 |
| A/B 测试生成 | ❌ | 同一章节生成 2 版，用户选优 |
| 生成质量评分 | ❌ | RAG 检索命中率自动统计 |
| Prompt 自动调优 | ❌ | 基于反馈数据自动优化 Prompt |
| 交叉学习 | ❌ | 租户 A 的优质语料（脱敏后）→ 全局知识 |

## 四、自动抓取招投标中心

### 📍 当前状态：❌ 未实现

### 🔧 可行性分析

| 维度 | 评估 |
|---|---|
| **技术可行性** | ✅ 高 — Python 爬虫生态成熟（Scrapy/Playwright） |
| **法律风险** | ⚠️ 中 — 需确认各平台 robots.txt 和用户协议 |
| **数据价值** | ✅ 高 — 招标公告/中标公示/评分标准 是核心训练数据 |
| **实现复杂度** | 🟡 中 — 各地平台格式不统一，需要适配器模式 |

### 🏗 建设方案（建议 v0.5 迭代）

```
【可抓取的数据源】
├── 全国公共资源交易平台（统一接口）
├── 各省市公共资源交易中心
├── 中国招标投标公共服务平台
└── 行业垂直平台（建设工程招投标网等）

【可抓取的内容类型】
├── 招标公告 → 提取评分标准模板
├── 中标公示 → 学习中标价格/团队配置规律
├── 招标文件附件 → 扩充知识库
└── 行业动态/政策 → 更新规范引用

【架构设计】
                    ┌─ 调度器 (APScheduler)
                    │    · 每日定时抓取
                    │    · 增量更新（防重复）
                    ↓
  数据源适配器       →  内容清洗管道  →  知识库入库
  (各平台接口)           · 去HTML       · 自动切片
                         · 提取正文     · 向量化
                         · 分类标签     · 租户:public

【伦理与合规红线】
├── ✅ 只抓取公开信息（招标公告、中标公示）
├── ✅ 遵守 robots.txt
├── ✅ 控制抓取频率（≤1次/秒，带 User-Agent）
├── ❌ 不抓取需要登录才能访问的内容
└── ❌ 不存储/传播涉及商业秘密的投标文件
```
