## 二、合规审查（Step 4）—— 需要建设什么

### 2.1 格式合规

标书有严格的格式规范，当前导出引擎做了基础格式，但还缺：

| 检查项 | 当前 | 应达到 |
|---|---|---|
| A4 纸张 + 标准页边距 | ✅ 已有 | — |
| 封面（项目名+投标人+日期） | ✅ 已有 | 需增加日期、密封条 |
| 目录自动生成 | 🟡 占位 | 需要真正的 Word 域目录 |
| 正文字体：仿宋_GB2312 / 12pt | ✅ 已有 | — |
| 标题字体：黑体 / 16-13pt 层级 | ✅ 已有 | — |
| 首行缩进两格 | ✅ 已有 | — |
| 行距 28 磅 | ✅ 已有 | — |
| 页码（页脚居中） | ✅ 已有 | — |
| 页眉（项目名） | ✅ 已有 | — |
| **表格规范** | ❌ 缺失 | 三线表/全框线、表头加粗、表注格式 |
| **图片插入规范** | ❌ 缺失 | 图序、图名、分辨率 ≥300dpi |
| **页码连续性** | ❌ 未验证 | 封面/目录/正文分节编页 |
| **字数统计** | ❌ 缺失 | 各章节字数是否达到评分最低要求 |
| **编号连续性** | ❌ 缺失 | 一、二、三...不跳号不重号 |

### 2.2 内容合规

| 检查项 | 说明 | 当前 |
|---|---|---|
| **响应性检查** | 每个评分点是否都有对应章节响应 | ❌ 缺失 |
| **关键词覆盖** | 招标文件的技术要求关键词是否在标书中出现 | ❌ 缺失 |
| **禁止条款** | 是否包含招标文件明确禁止的内容 | ❌ 缺失 |
| **数据一致性** | 工期/造价/人数等数据全文是否一致 | ❌ 缺失 |
| **规范引用** | GB/JTG 编号是否正确、是否已废止 | ❌ 缺失 |
| **公司信息** | 投标人名称/资质等级全文是否一致 | ❌ 缺失 |

### 2.3 建设方案

```python
# 建议新增模块：app/services/compliance_service.py

class ComplianceService:
    """合规审查服务"""

    def check_format(self, sections: dict) -> list[ComplianceIssue]:
        """格式规范检查"""
        # 1. 编号连续性检查
        # 2. 字数统计（对比评分要求的最低字数）
        # 3. 表格/图片格式校验

    def check_content(self, sections: dict, requirements: dict) -> list[ComplianceIssue]:
        """内容合规检查"""
        # 1. 评分点 → 章节响应性矩阵（是否每个评分点都覆盖）
        # 2. 关键词覆盖率（招标文件技术要求 vs 投标响应）
        # 3. 数据一致性校验（工期/金额/人数全文一致）
        # 4. 规范编号有效性（是否引用了已废止的标准）

    def check_disqualification_risks(self, sections: dict) -> list[ComplianceIssue]:
        """废标风险检查"""
        # 1. 投标人名称一致性
        # 2. 是否包含禁止性承诺
        # 3. 密封/签章提醒
```

## 三、多轮润色（Step 5）—— 需要建设什么

### 3.1 当前 vs 目标

```
当前流程（单轮）：
AI 生成 → 人工看一遍 → 采纳/修改 → 结束

目标流程（多轮迭代）：
AI 生成（初稿）
    ↓
Round 1: AI 自动润色（术语规范化 + 文风统一）
    ↓
Round 2: 合规审查（格式+内容检查 → 标注问题）
    ↓
Round 3: 人工审阅（修改/确认）
    ↓
Round 4: AI 精修（根据人工反馈二次优化）
    ↓
定稿 → 反审标 → 变体 → 导出
```

### 3.2 AI 润色需要做什么

| 润色层级 | 具体内容 | 举例 |
|---|---|---|
| **L1 术语规范** | 统一专业术语 | "水泥砂浆" vs "水泥沙浆" → 统一 |
| **L2 文风统一** | 全文语气、人称、时态一致 | "我公司" vs "本公司" → 统一 |
| **L3 逻辑增强** | 补充过渡句、因果关系 | 段落间加"基于上述分析..." |
| **L4 专业深化** | 增加数据支撑、规范引用 | 加入具体参数、标准编号 |
| **L5 亮点提炼** | 突出技术创新、差异化优势 | "采用BIM技术实现全过程管控" |

### 3.3 建设方案

```python
# 建议新增模块：app/services/polish_service.py

class PolishService:
    """多轮润色服务"""

    async def polish_terminology(self, content: str, glossary: dict) -> str:
        """L1 术语规范化：统一用词"""

    async def polish_style(self, content: str, style_guide: str) -> str:
        """L2 文风统一：调整语气、人称、时态"""

    async def polish_logic(self, content: str) -> str:
        """L3 逻辑增强：补充过渡、因果论述"""

    async def polish_professional(self, content: str, knowledge_ctx: str) -> str:
        """L4 专业深化：增加数据、规范引用"""

    async def polish_highlight(self, content: str, scoring_points: dict) -> str:
        """L5 亮点提炼：针对高分值评分点强化论述"""

    async def full_polish_pipeline(self, content: str, rounds: int = 3) -> PolishResult:
        """完整润色管道：L1→L5 逐级润色"""
```

---

> **结论**：当前系统在 **生成→检测→变体** 主链路上是通的，但在"质量保障"维度（合规审查 + 多轮润色 + Word 高级排版）存在明显缺口。这两个模块建议作为 **v0.4 版本的核心建设目标**。
